# 远方智设标题设计引擎 v1

## 标题系统目标升级

远方智设标准模式的标题层，不再定义为“字体排版”或“标题放置”，而是：

Reference-driven Multi-candidate Vector Title Composer

参考驱动的多候选向量标题设计系统。

旧方向包括：

- 字体库
- `titleArtStyle`
- `titleDirector`
- 横排 / 竖排 / 斜排
- Sharp 文字合成

这些只是技术底座，不是最终产品目标。标题不是普通文字叠加，而是视觉资产。最终目标是让标题接近专业设计师做过的主视觉标题，而不是“背景图上放几个中文字”。

当前判断：

- 背景层已经基本可用，目标约 75-80 分。
- 当前标题层只有约 15 分，不能继续凑合。
- v1.0 的目标不是先做 50 分可用版，而是冲 85 分。

系统边界：

- AI 不能直接生成中文标题图片。
- AI 只能输出结构化标题设计方案。
- 中文标题文字必须由系统使用真实字体 / glyph path / SVG / Sharp 稳定渲染。
- 所有标题字组拼接后必须等于原始 `mainTitle`，不能增字、漏字、改字。
- 标题方案必须受品牌 VI 控制，不能只追求炫技。

## v1.0 质量目标

- 背景层目标：75-80 分
- 标题层目标：85 分
- 不接受 50 分过渡方案
- 不以“能跑通”为完成标准
- 以“显著接近专业设计师标题主视觉”为完成标准

v1.0 的验收重点不是“是否能把字放上去”，而是标题是否像被设计过：有主次、有节奏、有形体、有装饰逻辑、有背景融合、有品牌一致性，并且中文内容 100% 准确。

## 标题系统核心架构

1. Title Reference Pattern Library

   标题参考模式库，从历史设计和优秀案例中抽象标题结构。参考模式记录标题重心、字组拆分、装饰位置、主副层级、留白关系、适用场景和禁用场景。

2. Title Candidate Generator

   一次生成 6-12 个标题候选，而不是单次决策。候选之间必须在构图、重心、字组节奏、装饰策略或背景融合方式上有真实差异。

3. Title Unit Composition

   字组级构图，把主标题拆成 `lead` / `main` / `accent` / `support` 等字组。拆分必须可逆，所有字组按顺序拼接后必须严格等于原始 `mainTitle`。

4. Candidate Preview Renderer

   先渲染小图候选，不直接进入最终成图。预览图用于快速比较标题可读性、视觉重心、背景融合和品牌气质。

5. Candidate Scorer

   用规则评分 + AI 视觉评审选择最佳标题方案。规则评分检查文字准确性、可读性、对比度、VI 色彩、构图安全区和候选多样性；AI 视觉评审只评价设计效果，不改写文字。

6. Vector Glyph Renderer

   使用真实字体转 glyph path / SVG path，使中文字成为可控向量图形资产。系统可以对字形做描边、阴影、渐变、装饰和组合，但不能让 AI 直接画中文。

7. Final Composer

   将最佳标题方案高清合成到背景图上。最终合成需要保留文字准确性、品牌 VI、背景可读性和输出稳定性。

## 标题设计约束

标题设计必须包含：

- 参考模式库
- 多候选生成
- 字组级构图
- 自动评分
- 向量字形渲染
- 背景融合
- 品牌 VI 控制

AI 的职责是理解活动目标、背景构图和参考模式，然后输出结构化标题设计方案。系统的职责是用真实字体和可控渲染链路把中文标题稳定生成出来。

任何标题候选都必须通过文字完整性校验：

- 不能增字
- 不能漏字
- 不能改字
- 不能重排到改变语义
- `lead + main + accent + support` 等字组拼接后必须等于原始 `mainTitle`

## 禁止路线

- 不把标题层降级为普通 SVG text 排版
- 不继续围绕固定坐标调参
- 不把横排 / 竖排 / 斜排当成标题设计本身
- 不让 AI 直接生成中文标题图片
- 不允许 AI 改写标题文字
- 不允许增字、漏字、错字
- 不用来源不明字体合集
- 不把字体文件提交到 GitHub
- 不把字体作为下载资源提供给用户
- 不让老师手动选择复杂字体参数
- 不为了艺术效果牺牲中文准确性和可读性

## 施工顺序

新的施工顺序：

1. 先更新 L2 目标文件
2. 建 Title Reference Pattern Library
3. 建 Title Candidate Generator
4. 建 Candidate Preview Renderer
5. 建 Candidate Scorer
6. 建 Vector Glyph Renderer
7. 接入 Final Compose
8. 最后再考虑正式前端和产品化

特别强调：

- 不要先继续改 `image-compose`
- 不要先继续扩 `title-director` 的单次 decision
- 不要先继续调字体大小、坐标、横竖斜

这些都不是当前阶段主线。当前主线是把标题系统从“单次参数决策 + Sharp 叠字”升级为“参考驱动 + 多候选 + 可评分 + 向量渲染 + 最终合成”的设计系统。

## 与背景层的关系

标准模式背景层的目标是活动主视觉背景，不是完整海报。背景必须为标题系统提供可读区域、构图支点和主题氛围，但不能要求 AI 在背景里直接生成标题。

标题系统必须读取背景候选或背景元信息，判断：

- 标题区域亮度
- 背景复杂度
- 视觉焦点位置
- 主题元素分布
- 安全区和品牌信息区

标题与背景的融合不是把字放在空白处，而是让标题成为画面主资产，同时不破坏背景主题。

## 与字体库的关系

字体库仍然重要，但它只是原材料。字体文件用于本地稳定渲染，不代表标题设计完成。

字体使用规则：

- 只能使用授权来源明确的字体
- 字体文件放在本地生成环境
- 字体文件不提交 GitHub
- 字体文件不作为下载资源提供给用户
- 生产上线前必须人工复核字体授权

真正的标题设计来自参考模式、字组构图、向量字形处理、装饰逻辑、背景融合和品牌控制。

## AI 输出边界

AI 可以输出结构化标题设计方案，例如：

```json
{
  "referencePatternId": "stage-title-hero-01",
  "titleUnits": [
    { "role": "lead", "text": "暑假" },
    { "role": "main", "text": "文学表达" },
    { "role": "support", "text": "体验课" }
  ],
  "composition": {
    "layoutIntent": "center-weighted-hero",
    "emphasisRole": "main",
    "backgroundFusion": "soft-glow-panel"
  },
  "vectorStyle": {
    "fontKey": "sourceHanSansBold",
    "stroke": "brand-safe-light",
    "shadow": "stage-depth"
  }
}
```

AI 不可以输出最终中文标题图片，不可以决定修改 `mainTitle`，不可以绕过字组拼接校验。

## 产品目标

远方智设要达到：

- 背景有主题差异
- 标题像视觉资产
- 中文文字准确
- 品牌 VI 统一
- 老师无需懂设计
- 输出可直接发朋友圈、家长群、小红书、公众号、活动现场物料
- 用较低软件成本替代校区重复设计工作

最终状态不是“老师会调字体”，而是老师只填写设计需求，系统自动生成一张品牌统一、标题有设计感、信息准确、可以直接传播的成品图。
